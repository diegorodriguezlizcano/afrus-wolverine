# ISS-002 — PostgreSQL Multi-Tenant Database Schema and Migrations

## Issue Summary
Design the complete PostgreSQL multi-tenant schema for afrus-Wolverine using Prisma ORM.
Organization is the tenant unit. Implement Row Level Security (RLS) on all tenant-scoped tables.
Set up versioned Prisma migrations. Seed default pipeline stages, lost reasons, and a default admin user per organization.

---

- **Roadmap.md**: All 24 issues, ISS-002 in Phase 0. Lead PK is `email`. 5 tag types: `stage:`, `origin:`, `temp:`, `action:`, `sync:`. `organizations` stores `afrus_org_id` and `afrus_api_key` per org.
- **docker-compose.yml**: Already has `postgres:16-alpine` service with `postgres_data` volume, healthcheck, port 5432. **DO NOT replace this — augment only.**

## Required Deliverables

### 1. Prisma Schema (`prisma/schema.prisma`)
Define ALL entities:

#### `organizations` (tenant root — NO RLS, it's the tenant table itself)
```
id              UUID    PK default uuid()
name            String
afrus_org_id    String  @unique  // from afrus platform
afrus_api_key   String  // per-org API key for afrus API calls
created_at      DateTime @default(now())
updated_at      DateTime @updatedAt
```
Note: `organizations` has NO `organization_id` FK — it IS the tenant.

#### `users`
```
id              UUID    PK default uuid()
organization_id  UUID    FK → organizations.id  (NOT NULL)
email           String  @unique  // global unique, not per-org unique
name            String
role            Enum: ADMIN | SDR | VIEWER
password_hash   String
created_at      DateTime @default(now())
updated_at      DateTime @updatedAt
```
Unique constraint: (email) — global, not org-scoped (one person = one login).

#### `leads`
```
email           String  PK  // email is the PK, NOT afrus_lead_id
organization_id  UUID    FK → organizations.id  (NOT NULL)
afrus_lead_id   String? // optional afrus reference

// Pipeline state
stage           Enum: NEW | SCHEDULED | MET | QUALIFIED | PROPOSED | NEGOTIATING | FUTURE | WON | LOST
temperature     Enum: COLD | WARM | HOT  @default(COLD)

// Names and contact
first_name      String?
last_name       String?
phone           String?
phone2          String?

// Qualified data
need_confirmed     Boolean @default(false)
budget_confirmed   Boolean @default(false)
urgency_confirmed  Boolean @default(false)
deal_value         Decimal?

// Meeting fields
scheduled_at       DateTime?
scheduled_by       UUID?  FK → users.id
met_at             DateTime?
qualified_at       DateTime?
proposed_at        DateTime?
negotiating_since  DateTime?
won_at             DateTime?
won_by             UUID?  FK → users.id
deal_value_won     Decimal?

// Lost fields
lost_at            DateTime?
lost_by            UUID?  FK → users.id

// Future stage
next_contact_date  DateTime?
reactivated_at     DateTime?

// Origin
origin_id          UUID?  FK → origins.id

// Current stage entry (most recent transition)
stage_entered_at   DateTime @default(now())

created_at         DateTime @default(now())
updated_at         DateTime @updatedAt
```

#### `origins` (per organization)
```
id              UUID    PK default uuid()
organization_id  UUID    FK → organizations.id (NOT NULL)
name            String  // e.g. "Web Form", "Referral", "Event"
description     String?
created_at      DateTime @default(now())
updated_at      DateTime @updatedAt
@@unique([organization_id, name])
```

#### `tags`
```
id              UUID    PK default uuid()
organization_id  UUID    FK → organizations.id (NOT NULL)
lead_email      String  FK → leads.email (NOT NULL)  -- composite with org
tag_type        Enum: STAGE | ORIGIN | TEMP | ACTION | SYNC
tag_value       String  // e.g. "new", "cold", "meeting_scheduled"
created_at      DateTime @default(now())
updated_at      DateTime @updatedAt
@@unique([lead_email, organization_id, tag_type])
```
Note: Only one tag per type per lead (enforced by unique index + tag_type enum).

#### `lost_reasons` (per organization)
```
id              UUID    PK default uuid()
organization_id  UUID    FK → organizations.id (NOT NULL)
reason          String  // e.g. "No budget", "No urgency", "Went silent"
description     String?
is_active       Boolean @default(true)
created_at      DateTime @default(now())
updated_at      DateTime @updatedAt
@@unique([organization_id, reason])
```

#### `sync_tags` (per organization — maps Wolverine sync tag to afrus tag name)
```
id              UUID    PK default uuid()
organization_id  UUID    FK → organizations.id (NOT NULL)
tag_value       String  // Wolverine-side tag value (e.g. "hot-leads")
afrus_tag_name  String  // afrus-side tag to filter on (e.g. "Hot Leads")
description     String?
is_active       Boolean @default(true)
created_at      DateTime @default(now())
updated_at      DateTime @updatedAt
@@unique([organization_id, tag_value])
```

#### `stage_transition_log`
```
id              UUID    PK default uuid()
lead_email      String  FK → leads.email (NOT NULL)
organization_id UUID    FK → organizations.id (NOT NULL)  // denormalized for RLS
from_stage      String?
to_stage        String  // stage enum value as string
triggered_by    UUID    FK → users.id (NOT NULL)
metadata        Json?   // extra context (e.g. { "lost_reason": "No budget" })
created_at      DateTime @default(now())
```

#### `action_tag_log`
```
id              UUID    PK default uuid()
lead_email      String  FK → leads.email (NOT NULL)
organization_id UUID    FK → organizations.id (NOT NULL)
action_tag      String  // e.g. "meeting_scheduled"
alma_sequence   String? // ALMA sequence name triggered
status          Enum: TRIGGERED | COMPLETED | FAILED | NEEDS_HUMAN
alma_response   Json?
triggered_by    UUID    FK → users.id
triggered_at    DateTime @default(now())
completed_at    DateTime?
created_at      DateTime @default(now())
updated_at      DateTime @updatedAt
```

#### `sync_log`
```
id              UUID    PK default uuid()
organization_id UUID    FK → organizations.id (NOT NULL)
lead_email      String  FK → leads.email (NOT NULL)
direction       Enum: TO_AFRUS | FROM_AFRUS
status          Enum: PENDING | SYNCED | FAILED
payload         Json?
error_message   String?
synced_at       DateTime?
created_at      DateTime @default(now())
updated_at      DateTime @updatedAt
```

### 2. RLS Policies (Raw SQL Migrations)
Prisma does NOT natively support RLS. Implement as raw SQL migration files.
Run after each Prisma migration via `migration.engine.applySql`.

**Pattern**: Create a `wolverine` Postgres role for the application. RLS policies check `organization_id` against a session variable set at connection time.

```sql
-- Enable RLS on all tenant-scoped tables
ALTER TABLE leads ENABLE ROW LEVEL SECURITY;
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
ALTER TABLE tags ENABLE ROW LEVEL SECURITY;
ALTER TABLE origins ENABLE ROW LEVEL SECURITY;
ALTER TABLE lost_reasons ENABLE ROW LEVEL SECURITY;
ALTER TABLE sync_tags ENABLE ROW LEVEL SECURITY;
ALTER TABLE stage_transition_log ENABLE ROW LEVEL SECURITY;
ALTER TABLE action_tag_log ENABLE ROW LEVEL SECURITY;
ALTER TABLE sync_log ENABLE ROW LEVEL SECURITY;

-- Policy: users can only see rows matching their organization_id
-- Each table gets a policy like:
CREATE POLICY tenant_isolation_leads ON leads
  USING (organization_id = current_setting('app.current_org_id')::uuid);
```

**Tables requiring RLS**: `leads`, `users`, `tags`, `origins`, `lost_reasons`, `sync_tags`, `stage_transition_log`, `action_tag_log`, `sync_log`.
**Tables exempt**: `organizations` (tenant root), `prisma_migrations` (Prisma internal).

**IMPORTANT**: The Prisma connection string must include the `schema` parameter pointing to `public`. RLS bypasses: superuser (postgres) and table owner bypass RLS by default — configure `wolverine` app user without superuser privileges.

### 3. Versioned Migrations Directory (`prisma/migrations/`)
Use `prisma migrate dev --name init` to generate. Expected structure:
```
prisma/migrations/
  0000_init/
    migration.sql        -- created by Prisma
  0001_add_rls_policies/
    migration.sql        -- raw SQL RLS policies
```

After running `prisma migrate dev`, manually add a second migration folder for RLS.

### 4. Seed Script (`prisma/seed.ts`)
Seed data for each new organization (called when org is created):
- 8 default pipeline stages (NEW, SCHEDULED, MET, QUALIFIED, PROPOSED, NEGOTIATING, FUTURE, WON, LOST)
- 5 default lost reasons: "No budget", "No urgency", "Went silent", "Competitor chosen", "Timing not right"
- 3 default origins: "Web", "Referral", "Event"
- Default admin user (password: hashed "changeme" — must be changed on first login)

Seed file must use Prisma Client directly. Run with `npx prisma db seed`.

---

### 5. Prisma Configuration
Update `schema.prisma` with:
```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

Add to `package.json`:
```json
"prisma": {
  "seed": "ts-node prisma/seed.ts"
}
```

---

### 6. Database URL Environment Variable
Add to `.env.example`:
```
DATABASE_URL="postgresql://wolverine:wolverine_secret@postgres:5432/wolverine?schema=public"
```
